# Mirage-Sentinel 完整資料流

更新日期：2026-04-21

這份文件描述整個專案目前的資料如何流動：從外部請求進來、經過 gateway 與 Sentinel 判斷、進入真實銀行或 Mirage 欺敵路徑、最後寫入不同資料庫與 SOC。

## 1. 一句話總結

Mirage-Sentinel 的資料流可以分成三層：

1. **流量入口層**：FastAPI gateway 接收所有請求，抽取特徵並決定路由。
2. **業務/欺敵層**：正常請求送往 `vuln-bank-main`，可疑請求改走 Mirage 或假會話。
3. **鑑識/分析層**：事件寫入 `traffic_logs.db`，欺敵狀態寫入 `mirage_memory.db`，SOC 再從事件庫做可視化與回放。

## 2. 主要元件

- `main.py`
  - 主 gateway。
  - 負責分流、假會話處理、欺敵回應、事件寫入。
- `core/sentinel.py` / `model/ai_sentinel.py`
  - 做風險判斷與攻擊分類。
- `core/mirage.py`
  - 生成 Mirage 假回應。
- `core/deception_db.py`
  - 保存假帳戶、虛擬卡、假轉帳、Bill Payments、假會話記憶。
- `core/traffic_db.py`
  - 保存 traffic / attack / SOC 事件。
- `vuln-bank-main`
  - 真實銀行業務服務，正常流量的 upstream。
- `services/dashboard_service.py`
  - SOC 查詢與聚合服務。

## 3. 全局資料流圖

```mermaid
flowchart TD
    A[Client / Attacker / Browser] --> B[Gateway main.py]
    B --> C[Feature Extraction\nIP, headers, body, query, token]
    C --> D[Principal ID Derivation]
    D --> E{Fake Session?}
    E -- Yes --> F[Decrypt / Reuse Mirage Memory]
    F --> G[Return Deception Response]
    E -- No --> H{Sentinel / Model Risk Decision}
    H -- Normal --> I[Proxy to vuln-bank-main]
    H -- Suspicious --> J[Mirage.generate_fake_data()]
    J --> K[Write deception_memory.db]
    K --> G
    I --> L[Normal Business Response]
    G --> M[Write traffic_logs.db]
    L --> M
    M --> N[SOC dashboard_service.py]
    N --> O[Dashboard / Query / Replay]
```

## 4. 請求進入後的完整流程

### 4.1 入口接收

請求先進到 `main.py`，gateway 會先做以下事情：

- 讀取 method、path、query、body
- 收集 headers、content-type、authorization、referer、user-agent
- 抽取 client IP、device id、session chain id
- 產生毫秒級時間戳

這一層的目標是把原始 HTTP 請求標準化成可分析事件。

### 4.2 主體識別

gateway 會推導 `principal_id`，也就是「互動主體識別」。

目前它通常來自：

- `X-User-Id`
- CIF / customer id
- 或 fallback 成 `proxy:<client_ip>`

這個 `principal_id` 會一路傳給 Mirage、欺敵資料庫與 traffic 記錄。

### 4.3 假會話優先檢查

gateway 會先檢查請求是否帶有 `mirage_session_*` token。

如果命中：

- 不再走正常 upstream
- 直接從 `mirage_memory.db` 或假資料快取回應
- 同時更新互動次數、hits、最後存取時間

這表示：一旦攻擊者已被引導進 Mirage，後續互動會保持一致性。

### 4.4 風險判斷與路由決策

若不是假會話，gateway 會再做 Sentinel / AI 判斷：

- 正常：送往 `vuln-bank-main`
- 可疑：走 Mirage 欺敵回應

在目前實作中，Mirage 可以由 `generate_fake_data()` 產生對應端點的 JSON 回應，再由 gateway 套成最終 response。

## 5. 正常使用者資料流

### 5.1 Login / Banking / Virtual Card / Bill Payment

正常請求會進入 `vuln-bank-main`，例如：

- `/login`
- `/transfer`
- `/virtual-cards`
- `/bill-payments`

業務結果回來後，gateway 仍會把這次互動寫到 traffic log。

### 5.2 正常資料庫

正常路徑使用真實業務資料庫：

- PostgreSQL `vulnerable_bank`

這是正常用戶的真實資料來源，不應與 `mirage_memory.db` 混用。

## 6. Mirage 欺敵資料流

### 6.1 何時進入 Mirage

當請求被判定為可疑、或已有假會話時，就會進入 Mirage。

### 6.2 Mirage 做什麼

Mirage 會根據 endpoint 與 attack vector：

- 生成擬真的 JSON
- 產生 login / transfer / balance / virtual card / graphql 對應格式
- 盡可能讓攻擊者繼續互動，而不是立刻暴露破綻

### 6.3 Mirage 的資料來源

Mirage 回應來自兩種來源：

- 模型生成：`core/mirage.py`
- 欺敵記憶重用：`core/deception_db.py`

### 6.4 Mirage 的資料去向

成功生成後，會寫入：

- `deception_memory`
- `fake_accounts`
- `mirror_users`
- `fake_transactions`
- `fake_cards`
- `fake_card_transactions`
- `fake_bill_payments`

## 7. 端點資料流範例

### 7.1 Login 流程

1. 請求打到 gateway。
2. gateway 抽取 `principal_id` 與登入欄位。
3. 若判斷可疑，呼叫 Mirage 生成假登入資料。
4. 假帳密與假 account id 寫入 `fake_accounts` / `mirror_users`。
5. 回傳假的 session token。
6. SOC 收到對應事件。

### 7.2 Virtual Card 流程

1. 請求打到 `/api/virtual-cards/create`、`/api/virtual-cards` 或卡片操作端點。
2. gateway 判斷是否為假會話或 Mirage 路徑。
3. 若是，寫入 `fake_cards`。
4. 進一步的 fund / freeze / update-limit / transactions 都會對同一張假卡做狀態更新。
5. 查詢列表時回傳同一批假卡資料。

### 7.3 Bill Payments 流程

1. 請求進入 bill payment 端點。
2. Mirage 提供 bill categories / billers / payment history。
3. 若是 balance payment，會同步扣假餘額。
4. 寫入 `fake_bill_payments` 與對應假交易紀錄。

### 7.4 Transaction History 流程

1. 請求進入 transactions 端點。
2. Mirage 從 `fake_transactions` 讀出這個 principal 的交易歷史。
3. 回傳與前面假轉帳、假繳費一致的歷史資料。

## 8. 資料庫分工

### 8.1 `traffic_logs.db`

用途：

- 保存請求與回應事件
- 保存 attack details
- 提供 SOC 聚合與回放

性質：

- append-only
- 不能作為欺敵主流程的業務狀態來源

### 8.2 `mirage_memory.db`

用途：

- 保存欺敵狀態
- 保存假登入、假卡片、假交易、假繳費
- 保存 fake session token 與互動深度

性質：

- 可讀寫
- 是 Mirage 的主要狀態庫

### 8.3 PostgreSQL `vulnerable_bank`

用途：

- 保存真實用戶資料
- 提供正常用戶路徑

性質：

- 只屬於正常業務資料域
- 不應被 Mirage 欺敵資料直接污染

## 9. SOC 資料流

1. gateway 把請求/回應寫入 `traffic_logs.db`。
2. attack details 與事件欄位保留分流資訊。
3. `services/dashboard_service.py` 從 traffic DB 讀取資料。
4. 前端 dashboard 顯示事件列表、趨勢與回放資訊。

SOC 最重要的是能回答三件事：

- 這筆流量是不是攻擊？
- 最後有沒有進 Mirage？
- Mirage 回應是否成功維持互動？

## 10. 目前的資料流重點

- 正常流量：`client -> gateway -> vuln-bank-main -> traffic_logs.db -> SOC`
- 攻擊流量：`client -> gateway -> Mirage -> deception_memory.db -> traffic_logs.db -> SOC`
- 假會話流量：`client -> gateway -> deception cache/db -> traffic_logs.db -> SOC`

## 11. 你可以用這份文件檢查什麼

- 某個端點是走正常上游，還是走 Mirage。
- 一筆假登入資料後面有沒有接上假卡、假交易、假繳費。
- SOC 能不能看到該筆事件的路由結果與 deception mode。
- 兩個資料庫的職責有沒有混淆。
