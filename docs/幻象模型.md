# 幻象模型在這個專案做了什麼

更新日期：2026-04-19

本文件聚焦「模型層」本身，不重複說明整個 gateway 或 SOC 的所有功能。

## 1. 幻象模型的角色

幻象模型是 Mirage-Sentinel 的欺敵內容生成核心。它的任務不是做最準確的風險判斷，而是：

- 針對可疑請求生成看起來合理的金融 API 假回應
- 維持攻擊者互動連續性，延長停留時間
- 讓攻擊流量走欺敵資料域，而不是觸及真實資料路徑

## 2. 目前已做的模型能力

### 2.1 依端點生成不同格式的回應

模型會依 endpoint 套用不同 prompt 模板，而不是只回固定 JSON。

目前已覆蓋的常見類型包含：

- login/register
- dashboard/transactions
- transfer
- virtual card
- balance
- graphql
- profile/upload

模型位置：
- [core/mirage.py](../core/mirage.py)

### 2.2 支援兩種推理來源

目前實作為可切換的雙來源：

- Ollama（預設優先）
- HuggingFace（可用時做替代）

並包含：

- 多個 Ollama URL 候選重試
- 超時與錯誤處理
- 模型不可用時回傳失敗訊號供上層處理

### 2.3 嚴格 JSON 解析與清洗

模型輸出後會做 JSON 清洗與 best-effort 解析，處理常見噪音：

- markdown code fence
- 前後多餘文字
- 尾逗號格式錯誤

若最終無法解析為 JSON，會視為模型輸出不可用，避免把不合法內容直接回給前端。

## 3. 模型在主流程中的實際作用

1. gateway 判斷需要走 deception 時，呼叫 Mirage 產生回應。
2. Mirage 模型回傳 JSON payload（成功時）。
3. 上層會將 payload 寫入欺敵記憶，後續可被 fake session 重用。
4. SOC 端可從事件欄位看到該回應來自 mirage 路徑。

關聯位置：
- [main.py](../main.py)
- [core/deception_db.py](../core/deception_db.py)
- [core/traffic_db.py](../core/traffic_db.py)

## 4. 模型已帶來的實際價值

- 回應內容不再是單一固定模板，降低被快速識破的機率
- 不同端點可回應對應結構，提升擬真度
- 失敗時有明確降級路徑，避免整條欺敵鏈斷裂

## 5. 實際運作範例（對照程式碼）

以下是這個專案裡，幻象模型實際會怎麼被用到的幾個例子。

### 範例 1：登入請求觸發 Mirage 假回應

當 gateway 判定請求需要走欺敵路徑時，會先用 `generate_fake_data()` 產生一個登入類 JSON。

實際流程大致如下：

```python
fake_response = generate_fake_data(
	principal_id,
	endpoint="/login",
	attack_vector=attack_vector,
)

fake_response["username"] = fake_username
fake_response["account_number"] = fake_account_id
fake_response["balance"] = login_balance
fake_response["session_token"] = fake_session_token
```

這表示：
- 模型先決定回應內容的大方向
- gateway 再把這份內容套成可登入、可回放的假帳號狀態
- 後續同一攻擊者再進來時，可以拿到連貫的假登入結果

### 範例 2：虛擬卡建立後，模型回應會接上資料庫狀態

當攻擊者打 `/api/virtual-cards/create` 時，gateway 會把請求內容交給欺敵資料庫建立卡片，再回傳卡片細節。

```python
record_fake_card(
	client_ip,
	session_principal_id,
	card_number,
	card_holder,
	expiry,
	cvv,
	card_type,
	currency=card_currency,
	card_limit=card_limit,
	current_balance=0.0,
	is_frozen=False,
)

fake_response = {
	"status": "success",
	"message": "Virtual card created successfully",
	"card_details": {
		"id": created_card.get("id"),
		"card_number": created_card.get("card_number"),
		"balance": created_card.get("balance", 0.0),
		"currency": created_card.get("currency", card_currency),
	},
}
```

這裡的重點不是「模型自己算出所有卡片資料」，而是：
- 模型/欺敵層先決定要回什麼樣的卡片回應
- 真正的卡片狀態寫進 `mirage_memory.db`
- 後續查詢列表或交易歷史時，可直接讀到同一批假卡

### 範例 3：假會話命中時直接重用 Mirage 記憶

如果攻擊者後續帶著 `mirage_session_*` token 回來，系統會先查 fake session，再決定是否直接回假資料。

```python
fake_session = get_fake_session(fake_session_token)
if fake_session:
	cached_memory = get_memory(client_ip, session_principal_id)
	fake_response = cast(dict[str, object], cached_memory["payload"])
	fake_response["response_origin"] = "deception_db"
	fake_response["deception_mode"] = "db_reuse"
```

這代表模型不一定每次都重新生成；一旦有記憶，系統會優先重用已經建立過的 Mirage 狀態，讓前後互動保持一致。

### 範例 4：模型輸出失敗時會安全降級

如果 Ollama / HuggingFace 沒有回傳可解析 JSON，`generate_fake_data()` 會回傳 `None`，上層就會送出結構化錯誤回應，而不是讓整條流程炸掉。

```python
ai_data = _parse_json_dict(summary)
if ai_data is not None:
	payload.update(ai_data)
	return payload

return None
```

這樣做的目的：
- 保持 API 可用
- 避免前端收到亂碼或半成品內容
- 讓假回應失敗時仍然有可觀測事件

## 6. 目前邊界

- 幻象模型負責「生成與擬真」，不負責最終是否攔截的策略決策。
- 最終分流仍由 gateway + 風險判斷控制。
- 模型輸出必須可解析為 JSON 才會進入正式回應。

## 7. 快速檢查清單

你可以用這份清單確認模型目前是否正常運作：

- 同一類端點是否回傳對應格式（不是 generic 固定回應）
- 模型失敗時是否有安全降級，而非直接錯誤崩潰
- 回應是否被寫入 deception 記憶並可被後續命中重用
- 事件中是否能辨識 response_origin 為 mirage 路徑
